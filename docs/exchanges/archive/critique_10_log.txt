## Iteration 10 Critique — Key Takeaways

### The One Big Thing
**Label-aware `check_pair` — re-run all three conditions (A/B/C).** The entire comparison table (A=0.51, B=0.20, C=0.55) is measured under a proxy condition (`>= 1 instance`) rather than Task 1's actual condition (`numeric_value OR location`). This is ~15 lines of code, ~$5, and determines whether the paper's central claim applies to the real task or just the proxy.

### New Issues Found (not previously raised)

1. **Pruning telemetry is lying.** The `f1_progression_results_iter9.json` token data *proves* pruning fired between turns 2→3 (tokens dropped 9,626→3,845, which is impossible without history compression), yet `prune_count=0` for all turns. The Iteration 9 attribute fix was likely applied to the code but not to the run that generated the results file. Must be verified with a direct `._prune_count` print on the next API run.

2. **Condition B token anomaly (3.4× unexplained cost).** Condition B uses 21,934 input tokens for 5K chars; Condition A Turn 1 uses 6,401 for the same 5K chars. The hypothesis is that Condition B runs all 6 max_iterations while Condition A terminates early (~2 iterations). This needs to be confirmed with iteration-count logging — and if true, it's a publishable finding about protocol efficiency.

3. **`_extract_assigned_names` nested-scope bug.** `ast.walk(tree)` descends into function bodies, so variables assigned inside model-written helper functions appear as module-scope variables in pruned history summaries. Fix: use `tree.body` iteration instead.

### Architectural Finding to Publish
The token data reveals that **pruning fires yet compliance holds (100%)** — because the deduplication guard operates on REPL Python state, not message history. This is a publishable architectural insight: REPL-persistent state decouples computational correctness from conversational context. History can be aggressively pruned without risking correctness. This should be stated explicitly in the paper.
