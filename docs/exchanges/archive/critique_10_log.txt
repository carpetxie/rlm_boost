## Key Findings

**What's Strong:**
- The Condition D fair comparison (77-86% token savings at 100% quality across 3 tasks) is the paper's strongest result and passes the 3rd-party clarity test cleanly
- Multi-run stability is publication-grade (σ=0.004 at k=5, σ=0.000 at k=3)
- The core `incremental.py` code is well-implemented with no new bugs found
- At-risk fraction predictor is validated across 3 tasks

**The One Big Thing:**
- **Run the dynamic context proof-of-concept** — deferred for 3 iterations, but all higher-priority blockers are now resolved. The retraction mechanism (the most novel architectural component) has never been tested with actually-changing data in a live API run. A $1-2 experiment would transform the retraction taxonomy from "simulation finding" to "empirically demonstrated."

**Other Key Issues:**
1. Token savings reporting should use the **structural formula** (deterministic: 66.7% at k=5) as primary metric, since D's token counts vary 3× across runs due to stochastic iteration counts
2. The "Dynamic RLM" framing has zero supporting live experiments — paper must either run the dynamic context experiment or reframe as "Incremental Sequential Processing"
3. A full-corpus (96K) experiment would remove the "artificial 25K budget" reviewer concern
4. Oracle C has stochastic failures (Task 6 Run 2: F1=0.0) — needs retry logic
