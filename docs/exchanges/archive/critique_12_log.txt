## Critique Summary — Iteration 12

**Scores:** Novelty 7/10 (+0), Technical Soundness 7/10 (+1), Benchmark Performance 7/10 (+1), Scalability 6/10 (+0), Research Maturity 7/10 (+0)

### Central New Finding: Attribute-Overwriting Bug

The most critical issue identified is a **correctness bug in the REPL template code** (`CHUNK_PROMPT_LABEL_AWARE_V2`) that directly undermines the paper's architectural claim about "qualification-time asymmetry":

The `entities` dict is rebuilt from scratch each chunk. When user X was qualifying in chunk 0 but reappears in chunk 2 with only non-qualifying labels, the REPL sets `entities[X] = {qualifying: False}` and `EntityCache.add()` **overwrites** X's qualifying=True with qualifying=False — triggering incorrect retractions and losing X's pairs permanently.

For "at least one" tasks (Tasks 1, 3, 6), once qualifying, always qualifying. The fix is 2 lines. **This bug may explain much of the 663 missing pairs (40.1% gap from oracle ceiling)**, making the "qualification-time asymmetry" framing potentially incorrect.

### The One Big Thing

**Run the attribute-overwriting ablation (~$3, 2 hours) before implementing lazy evaluation.** If A/C ratio increases toward 90%+, the paper's headline claim strengthens dramatically and the architectural narrative changes. If A/C is unchanged at ~64%, the asymmetry framing is confirmed and lazy eval is the right next step. Either outcome is publishable — but don't spend money on lazy eval before knowing which world you're in.

### Other Key Points
- **Token cost transparency**: Condition A uses 14% *more* LLM tokens than oracle C — this must be stated clearly; don't claim token savings
- **Task 6 A/C gap (55.5% vs 64%)**: unexplained 9pp difference, needs a per-chunk qualifying entity distribution analysis
- **k-sensitivity** (researcher's proposed priority 2): endorsed, but run after the ablation
- **5 code issues** identified, including double-counting in `updated × all` sweep and wrong system prompt for Condition B
